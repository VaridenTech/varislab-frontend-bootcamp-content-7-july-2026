จุดอ่อนของ JavaScript ที่ TypeScript เข้ามาแก้

ก่อนจะเริ่มเขียน TypeScript เรามาย้อนดูกันก่อนว่าปัญหาที่ TypeScript ถูกสร้างมาแก้จริง ๆ แล้วหน้าตาเป็นอย่างไร แล้วค่อยติดตั้ง TypeScript เพื่อลองดักจับปัญหาเหล่านั้นด้วยตัวเอง

1. Dynamic Typing: Type เปลี่ยนได้ตลอดเวลา

ใน JavaScript ตัวแปรหนึ่งตัวสามารถเก็บค่าคนละ Type กันได้ในเวลาต่างกัน ปัญหาจะเกิดขึ้นเมื่อโค้ดส่วนหนึ่งคาดหวังว่าตัวแปรจะเป็น Type หนึ่ง แต่ดันได้อีก Type มาแทน 

let price = 129.99;
price.toFixed(2); // "129.99" ใช้ได้ปกติ

price = "sold out";
price.toFixed(2); // TypeError: price.toFixed is not a function

โค้ดสองบรรทัดแรกไม่มีปัญหาอะไรเลย แต่พอ price ถูกเปลี่ยนเป็น String ในภายหลัง โค้ดที่เคยทำงานถูกต้องก็พังทันที และ JavaScript จะไม่เตือนอะไรเลยจนกว่าโค้ดบรรทัดนั้นจะถูกรันจริง

2. Weak Typing: การแปลง Type แบบเดาใจ (Coercion)

JavaScript ยังพยายาม "ช่วยเหลือ" เราด้วยการแปลง Type ให้อัตโนมัติเมื่อเจอ Operator ที่ไม่ตรงกับ Type ผลลัพธ์ที่ได้บางครั้งก็ไม่ตรงกับที่คาดไว้เลย 

const total = "20" + 5;      // "205"  (บวกเลขเข้ากับ String กลายเป็นต่อ String)
const discount = "20" - 5;   // 15    (ลบ String จะถูกแปลงเป็นตัวเลขก่อน)
const combined = [] + {};    // "[object Object]"

ผลลัพธ์เหล่านี้ไม่ใช่ Bug ของ JavaScript แต่เป็นพฤติกรรมที่ถูกออกแบบไว้ตั้งแต่แรก เพียงแต่มักไม่ตรงกับสิ่งที่นักพัฒนาตั้งใจเขียน

ทำไมปัญหานี้ถึงอันตราย

จุดร่วมของทั้งสองปัญหาคือ ข้อผิดพลาดจะถูกพบก็ต่อเมื่อโค้ดบรรทัดนั้นถูกรันจริงเท่านั้น ถ้าโค้ดส่วนนั้นไม่ค่อยถูกเรียกใช้ (เช่น กรณีขอบ หรือ Path ที่ผู้ใช้ไม่ค่อยเดิน) Bug อาจซ่อนอยู่ในโปรเจกต์เป็นเดือนโดยไม่มีใครรู้ จนกระทั่งผู้ใช้งานจริงไปเจอเข้า

คำถามคือ — เราจะรู้ได้อย่างไรว่าโค้ดของเรามีจุดที่เสี่ยงแบบนี้อยู่ ก่อน ที่จะรันโค้ดจริง? นี่คือสิ่งที่ TypeScript เข้ามาช่วย

ติดตั้ง TypeScript

ติดตั้ง TypeScript แบบ Global ผ่าน npm เพื่อใช้คำสั่ง tsc (TypeScript Compiler) จาก Terminal ได้ทุกที่:

npm i -g typescript

ตรวจสอบว่าติดตั้งสำเร็จ:

tsc --version

ให้ TypeScript ตรวจไฟล์ .js ของเราเอง

สิ่งที่หลายคนไม่รู้คือ tsc ไม่จำเป็นต้องใช้กับไฟล์ .ts เท่านั้น เราสามารถสั่งให้มันตรวจสอบไฟล์ .js ธรรมดาได้ด้วย Flag --checkJs โดยที่ยังไม่ต้องแปลงไฟล์เป็น TypeScript เลย

สมมติเรามีไฟล์ price.js ที่มีโค้ดจากตัวอย่างด้านบน:

tsc --checkJs --noEmit price.js
Flag ความหมาย
--checkJs สั่งให้ตรวจสอบ Type ในไฟล์ .js ด้วย (ปกติ tsc จะดูแค่ไฟล์ .ts)
--noEmit แค่ตรวจสอบ Type เท่านั้น ไม่ต้องสร้างไฟล์ผลลัพธ์ออกมา

ลองเพิ่ม Flag --strict เข้าไปอีกตัว เพื่อให้ TypeScript ตรวจสอบแบบเข้มงวดขึ้น (จะพบปัญหาเพิ่มอีกหลายจุดที่ก่อนหน้านี้ไม่ถูกจับได้เลย): 

tsc --checkJs --noEmit --strict price.js

สรุป

แม้จะยังไม่ได้เขียน Type ใด ๆ เองเลยสักตัว แค่ปล่อยให้ tsc อ่านไฟล์ JavaScript เดิมของเราด้วย --checkJs ก็ช่วยเผยปัญหาที่ซ่อนอยู่ได้แล้ว บทถัดไปเราจะเริ่มเขียน Type Annotation ด้วยตัวเอง เพื่อบอก TypeScript อย่างชัดเจนว่าตัวแปรแต่ละตัวควรเป็น Type อะไร
