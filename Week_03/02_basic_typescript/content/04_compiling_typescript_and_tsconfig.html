การ Compile TypeScript และไฟล์ tsconfig.json

Browser และ Node.js ไม่รู้จักไวยากรณ์ของ TypeScript เลย — สิ่งที่รันได้จริงมีแค่ JavaScript เท่านั้น ดังนั้นก่อนโค้ด .ts ของเราจะถูกใช้งานจริง มันต้องผ่านขั้นตอน Compile เพื่อแปลงเป็น .js ก่อนเสมอ

tsc แปลง .ts เป็น .js อย่างไร

ลองดูตัวอย่างไฟล์ TypeScript นี้:

// greet.ts
function greet(name: string): string {
  return `สวัสดี ${name}`;
}

เมื่อสั่ง Compile:

tsc greet.ts

TypeScript จะสร้างไฟล์ greet.js ขึ้นมาใหม่ โดยมีเนื้อหาประมาณนี้:

// greet.js
function greet(name) {
  return `สวัสดี ${name}`;
}

สังเกตว่า Type Annotation ทั้งหมด (: string) หายไป จากไฟล์ผลลัพธ์ นี่คือหลักการสำคัญของ TypeScript: Type มีไว้ช่วยตรวจสอบตอนเขียนโค้ดเท่านั้น เมื่อ Compile เสร็จแล้ว Type ทั้งหมดจะถูกลบทิ้ง เหลือแค่ JavaScript ธรรมดาที่พร้อมรันจริง

ทบทวน Flag ที่ใช้บ่อย

Flag ทำหน้าที่อะไร
--checkJs ให้ตรวจสอบไฟล์ .js ด้วย (ปกติดูแค่ .ts)
--noEmit ตรวจสอบ Type อย่างเดียว ไม่สร้างไฟล์ .js ออกมา
--strict เปิดโหมดตรวจสอบแบบเข้มงวดที่สุด
--target <version> กำหนดว่า JavaScript ที่ Compile ออกมาต้องรันได้บน Environment รุ่นไหน

Compiler Target

Flag --target บอก TypeScript ว่า JavaScript ที่จะสร้างออกมาต้องรองรับ Environment เก่าแค่ไหน 

tsc --target es2020 greet.ts
Target เหมาะกับ
es5 Browser รุ่นเก่ามาก (ค่าเริ่มต้นของ TypeScript)
es2015 / es6 Browser ส่วนใหญ่ในปัจจุบัน
esnext Environment ล่าสุดที่รองรับ JavaScript feature ใหม่ที่สุด

ไฟล์ tsconfig.json: ตั้งค่าครั้งเดียว ใช้ได้ทั้งโปรเจกต์

การพิมพ์ Flag ยาว ๆ ทุกครั้งที่ Compile นั้นไม่สะดวก โปรเจกต์จริงจึงเก็บค่าตั้งค่าเหล่านี้ไว้ในไฟล์ tsconfig.json แทน

สร้างไฟล์ตั้งต้นได้ด้วยคำสั่ง:

tsc --init

ตัวอย่างไฟล์ tsconfig.json แบบเรียบง่าย:

{
  "compilerOptions": {
    "target": "es2020",
    "strict": true,
    "outDir": "dist",
    "rootDir": "src"
  }
}

เมื่อมีไฟล์นี้อยู่ในโปรเจกต์แล้ว การรัน tsc เฉย ๆ (ไม่ต้องระบุชื่อไฟล์หรือ Flag ใด ๆ) ก็จะใช้ค่าตั้งค่าจากไฟล์นี้โดยอัตโนมัติ

ตัวเลือกที่พบบ่อยเพิ่มเติม

ตัวเลือก สร้างไฟล์อะไร ใช้ทำอะไร
declaration .d.ts ให้ Editor แนะนำ Type ของโค้ดเราตอนถูก Import ไปใช้ที่อื่น
sourceMap .js.map ให้ DevTools แสดงตำแหน่งอ้างอิงกลับไปยังไฟล์ .ts ต้นฉบับตอน Debug

Interface: อีกวิธีในการกำหนดรูปร่างของ Object

นอกจาก type ที่เรียนไปแล้ว TypeScript ยังมี Keyword interface สำหรับกำหนดรูปร่างของ Object โดยเฉพาะ: 

interface User {
  username: string;
  id: number;
}

const currentUser: User = { username: "somchai", id: 1 };

จุดเด่นของ interface คือขยาย (Extend) จาก Interface อื่นได้ตรง ๆ ด้วย Keyword extends:

interface Customer extends User {
  loyaltyPoints: number;
}

interface Admin extends User {
  permissionLevel: "staff" | "owner";
}

const jane: Customer = { username: "jane", id: 2, loyaltyPoints: 120 };   // OK
const noPoints: Customer = { username: "kim", id: 3 };                     // Error! ขาด loyaltyPoints

const root: Admin = { username: "root", id: 4, permissionLevel: "owner" }; // OK

ในทางปฏิบัติ type และ interface ทำงานคล้ายกันมากสำหรับกำหนดรูปร่าง Object ทีมส่วนใหญ่เลือกใช้อย่างใดอย่างหนึ่งให้สม่ำเสมอทั้งโปรเจกต์ — interface มักถูกเลือกเมื่อคาดว่า Type นั้นจะถูกขยายต่อในอนาคต

Type ของโค้ด Async

Function ที่เป็น async จะคืนค่าเป็น Promise เสมอ ไม่ใช่ค่าตรง ๆ — TypeScript จึงกำหนด Type ของค่าที่ Return โดยห่อด้วย Promise<...> 

function getGreeting(): string {
  return "สวัสดี";
}

async function getGreetingAsync(): Promise<string> {
  return "สวัสดี";
}

const message = getGreeting();          // string
const pendingMessage = getGreetingAsync(); // Promise<string>

message.toUpperCase();       // OK
pendingMessage.toUpperCase(); // Error! Promise ไม่มี Method toUpperCase

Promise<string> คือตัวอย่างของ Type Parameter — การระบุว่า "Promise นี้ เมื่อเสร็จแล้วจะให้ค่าชนิดอะไร"

Generics: เขียน Type ที่ใช้ซ้ำได้กับหลาย Type

แนวคิดเดียวกับ Type Parameter ของ Promise เราสามารถนำมาใช้กับ Type ของตัวเองได้ เรียกว่า Generics โดยใส่ตัวแปร Type (มักตั้งชื่อว่า T) ไว้ในเครื่องหมาย <> 

type Nullable<T> = T | null;

let maybeId: Nullable<number> = 4;
maybeId = null;   // OK
maybeId = "four"; // Error!

let maybeName: Nullable<string> = null;
maybeName = "พลอย"; // OK
maybeName = 5;        // Error!

Nullable<T> เขียนครั้งเดียว แต่ใช้ซ้ำได้กับ Type ใดก็ได้ที่ส่งเข้ามาแทน T

Function ก็เป็น Generic ได้เช่นกัน:

function firstItem<T>(items: T[]): T {
  return items[0];
}

firstItem([1, 2, 3]);          // Type ที่ได้กลับมาคือ number
firstItem(["a", "b", "c"]);    // Type ที่ได้กลับมาคือ string

TypeScript จะ "อนุมาน (Infer)" ว่า T คือ Type อะไรจากค่าที่ส่งเข้าไปจริงในแต่ละครั้งที่เรียกใช้ Function โดยอัตโนมัติ

Utility Types: Generic สำเร็จรูปที่มากับ TypeScript

TypeScript มี Generic Type สำเร็จรูปให้ใช้ได้เลยสำหรับดัดแปลง Type ที่มีอยู่แล้ว โดยไม่ต้องเขียนใหม่ทั้งหมด ตัวอย่างที่ใช้บ่อยที่สุด ได้แก่ 

interface Product {
  id: number;
  name: string;
  price: number;
}

Readonly<T> — ห้ามแก้ไข Property หลังกำหนดค่าแล้ว

const draft: Product = { id: 1, name: "เสื้อยืด", price: 259 };
draft.price = 299; // OK

const published: Readonly<Product> = { id: 2, name: "กางเกง", price: 590 };
published.price = 490; // Error! Cannot assign to 'price' because it is a read-only property

Partial<T> — ทำให้ทุก Property ไม่บังคับต้องมี

const productUpdate: Partial<Product> = { price: 199 }; // OK - ไม่ต้องใส่ id หรือ name ก็ได้

function updateProduct(id: number, changes: Partial<Product>) {
  // นำ changes ไปอัปเดตเฉพาะ Property ที่ส่งมา
}

Pick<T, Keys> — เลือกมาแค่บาง Property

type ProductSummary = Pick<Product, "id" | "name">;

const summary: ProductSummary = { id: 1, name: "เสื้อยืด" }; // OK, ไม่ต้องมี price

Omit<T, Keys> — ตัดบาง Property ออก

type NewProduct = Omit<Product, "id">; // Product แบบไม่มี id เพราะยังไม่ถูกสร้างในระบบ

const draftProduct: NewProduct = { name: "หมวก", price: 350 }; // OK

ยังมี Utility Type อื่นอีกมาก เช่น Record, Required, Exclude — ดูรายการทั้งหมดได้ที่ TypeScript Handbook: Utility Types

ทางลัดเมื่อจนตรอกจริง ๆ: any และ @ts-ignore

บางครั้งเราอาจเจอสถานการณ์ที่ยังหา Type ที่ถูกต้องไม่ได้ TypeScript มีทางลัดไว้ให้ แต่ควรใช้ให้น้อยที่สุด

let anything: any = "อะไรก็ได้";
anything = 5;
anything = { a: 1 };
anything.toUpperCase(); // ไม่มี Error เลย แม้ตอนรันจริงอาจพังก็ตาม

any คือการปิดการตรวจสอบ Type ของตัวแปรนั้นไปเลย — เหมือนกลับไปเขียน JavaScript ธรรมดา ใช้ได้ในกรณีจำเป็นจริง ๆ แต่ไม่ควรใช้พร่ำเพรื่อ เพราะเสียจุดประสงค์หลักของการใช้ TypeScript

อีกทางลัดหนึ่งคือ Comment พิเศษ @ts-ignore ซึ่งสั่งให้ปิดการตรวจสอบเฉพาะบรรทัดถัดไปเท่านั้น:

// @ts-ignore
const legacyResult = someUntypedLibraryFunction();

เก็บทางลัดทั้งสองนี้ไว้ใช้เฉพาะจุดที่จำเป็นจริง ๆ พร้อม Comment อธิบายเหตุผลกำกับไว้เสมอ เพื่อไม่ให้เพื่อนร่วมทีมงงว่าทำไมตรงนี้ TypeScript ถึงเงียบ
