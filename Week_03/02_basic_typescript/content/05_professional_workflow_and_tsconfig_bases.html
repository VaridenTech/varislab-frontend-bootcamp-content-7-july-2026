Workflow มืออาชีพ: ย้ายโปรเจกต์เดิมมาเป็น TypeScript

โปรเจกต์จริงในที่ทำงานแทบไม่มีใครเริ่มเขียนใหม่ทั้งหมดด้วย TypeScript ตั้งแต่วันแรก ส่วนใหญ่คือการค่อย ๆ ย้าย โปรเจกต์ JavaScript ที่มีอยู่แล้วมาเป็น TypeScript ทีละไฟล์ บทนี้จะพาดูขั้นตอนที่ทีมพัฒนาใช้กันจริงในการทำแบบนั้น

ขั้นตอนที่ 1: ติดตั้ง TypeScript เป็น Dev Dependency

ต่างจากบทก่อนที่เราติดตั้งแบบ Global เพื่อทดลองใช้ โปรเจกต์จริงควรติดตั้ง TypeScript ไว้ใน package.json ของโปรเจกต์เอง เพื่อให้ทุกคนในทีมใช้ TypeScript เวอร์ชันเดียวกัน 

npm i -D typescript

ขั้นตอนที่ 2: ตั้งค่า tsconfig.json ด้วย Base สำเร็จรูป

แทนที่จะเขียนค่าตั้งค่าใน tsconfig.json เองทั้งหมด ชุมชน TypeScript มีค่าตั้งค่าสำเร็จรูป (เรียกว่า "Base") สำหรับ Use Case ต่าง ๆ ให้เลือกใช้ได้เลย เช่น โปรเจกต์ที่ใช้ Node.js, โปรเจกต์ที่ใช้ Vite, โปรเจกต์ React

ติดตั้ง Base ที่เหมาะกับโปรเจกต์ (ตัวอย่างนี้ใช้ Base แนะนำทั่วไป):

npm i -D @tsconfig/recommended

แล้วให้ tsconfig.json ของเรา "สืบทอด" ค่าจาก Base นั้นด้วย Key extends:

{
  "extends": "@tsconfig/recommended/tsconfig.json",
  "compilerOptions": {
    "outDir": "dist",
    "rootDir": "src"
  }
}

ถ้าต้องใช้หลาย Base ร่วมกัน (เช่น Base ทั่วไป + Base เฉพาะของ Framework) ก็ระบุเป็น Array ได้:

{
  "extends": [
    "@tsconfig/recommended/tsconfig.json",
    "@tsconfig/vite-react/tsconfig.json"
  ]
}

อยากรู้ว่าค่าตั้งค่าสุดท้ายที่ TypeScript จะใช้จริง (หลังรวมทุก Base เข้าด้วยกันแล้ว) หน้าตาเป็นอย่างไร ใช้คำสั่ง:

tsc --showConfig

ขั้นตอนที่ 3: เปลี่ยนไฟล์ทีละไฟล์ ไม่ใช่ทีเดียวทั้งโปรเจกต์

สมมติเรามีโปรเจกต์ร้านค้าออนไลน์ขนาดเล็กที่เขียนด้วย JavaScript ล้วน: 

src/
  CartSummary.js
  ProductList.js
  formatCurrency.js

เริ่มจากไฟล์ที่มีความเสี่ยงต่ำและมีคนอื่นพึ่งพาน้อยที่สุดก่อน เช่น Utility Function ง่าย ๆ:

mv src/formatCurrency.js src/formatCurrency.ts

แค่เปลี่ยนนามสกุลไฟล์ (File Extension) ก็ถือว่าไฟล์นั้นเป็น TypeScript แล้ว (เพราะ TypeScript เป็น Superset ของ JavaScript) จากนั้นค่อยรัน tsc เพื่อดูว่ามี Error อะไรโผล่มาบ้าง แล้วค่อย ๆ เติม Type Annotation ทีละจุด 

tsc

ทำซ้ำแบบเดียวกันกับไฟล์ที่เหลือ ไล่จากไฟล์ที่ถูกพึ่งพาน้อยไปหามาก จนกว่าทั้งโปรเจกต์จะเป็น .ts ครบ

ผูก tsc เข้ากับ Workflow การพัฒนาประจำวัน

ตรวจ Type อัตโนมัติทุกครั้งที่บันทึกไฟล์

tsc --watch

คำสั่งนี้จะรัน TypeScript ค้างไว้เป็น Process เดียว แล้วตรวจ Type ใหม่ทุกครั้งที่มีไฟล์ถูกแก้ไข (ข้อควรระวัง: โปรเจกต์ใหญ่มากอาจทำให้ตรวจสอบช้าลง)

ให้การตรวจ Type เป็นด่านแรกก่อนรัน Test

ผูกคำสั่ง tsc --noEmit ไว้เป็นขั้นตอนแรกใน Script ทดสอบ เพื่อให้ Error ด้าน Type ถูกจับตั้งแต่ก่อน Test จะรันเลย:

{
  "scripts": {
    "test": "tsc --noEmit && vitest",
    "compile": "tsc"
  }
}

ด้วยรูปแบบนี้ ถ้า tsc --noEmit เจอ Error ด้าน Type คำสั่ง && จะทำให้ vitest ไม่ถูกรันต่อเลย — ทีมจะไม่มีทางเห็น Test ผ่านทั้งที่โค้ดมี Type ผิดอยู่

สรุป Checklist การย้ายโปรเจกต์มาเป็น TypeScript

  1. ติดตั้ง typescript เป็น Dev Dependency ของโปรเจกต์
  2. เลือก tsconfig Base ที่ตรงกับลักษณะโปรเจกต์ แล้ว extends จากมัน
  3. เปลี่ยนไฟล์เป็น .ts ทีละไฟล์ เริ่มจากไฟล์ที่ถูกพึ่งพาน้อยที่สุด
  4. รัน tsc หลังเปลี่ยนแต่ละไฟล์ แล้วแก้ Error ที่โผล่มาก่อนไปไฟล์ถัดไป
  5. ผูก tsc --noEmit เข้ากับ Script ทดสอบ เพื่อกันไม่ให้ Type Error หลุดเข้า Production
